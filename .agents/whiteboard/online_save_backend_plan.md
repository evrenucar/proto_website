# Online Whiteboard Save Plan (Backend / GitHub PR Flow)

# for now this plan is no longer necessary

## Purpose

This plan adapts the current Braindump whiteboard into a reusable system that can appear on multiple pages and let visitors submit changes to GitHub as recommendations.

The plan also corrects the GitHub flow in the earlier draft: a public visitor using their own GitHub account generally cannot create branches directly in the upstream repository unless they already have write access. For non-collaborators, the practical GitHub-native flow is:

1. authenticate with GitHub
2. create or reuse the visitor's fork
3. create or reuse a branch in that fork
4. commit the updated board file there
5. open or reuse a pull request back to the upstream repo

That aligns with "send as a recommendation" much better than pretending the change is already accepted.

## Current Repo Reality

The current repo is a static site generated by `scripts/build-site.mjs`, with page metadata in `src/site-data.mjs` and a GitHub Action that rebuilds generated content on push.

The whiteboard exists today as one special page:

- `src/site-data.mjs` defines the Braindump page metadata.
- `scripts/build-site.mjs` injects a page-specific whiteboard shell and toolbar.
- `JavaScript/braindump.js` contains all whiteboard behavior.
- `CSS/braindump.css` styles the page.
- `content/braindump-state.json` is the current canonical board file.
- `scripts/dev-server.mjs` exposes `POST /api/save-braindump`, but only for localhost editing.

The whiteboard is currently:

- page-specific, not reusable across multiple pages
- local-first, with `localStorage` key `braindump-canvas`
- optionally repo-saving only when the local dev server is running
- exporting with a hardcoded filename `braindump.canvas`

That means external visitors cannot actually save back to GitHub today.

## Logic Review Of The Existing Draft

### What still makes sense

- Static frontend plus a tiny token exchange backend is a good shape for a GitHub Pages site.
- GitHub OAuth is a reasonable way to let users submit under their own account.
- Using the GitHub REST API for forks, file writes, and pull requests is the right integration surface.
- Octokit is a reasonable client library if the API calls stay in frontend code.

### What does not make sense in the current draft

The draft assumes:

- user logs in with GitHub
- frontend creates a branch in `YOUR_REPO_NAME`
- frontend commits into that branch
- frontend opens a PR into `main`

That only works if the authenticated user already has write access to your repository.

For normal public site visitors, the correct flow is:

- user logs in
- app creates or reuses the user's fork of your repo
- app creates or reuses a recommendation branch in the fork
- app commits the board file in the fork
- app opens a PR from `userfork:branch` to `upstream:main`

### Recommendation wording

Do not label this feature "Save to GitHub".

Use wording like:

- `Send recommendation`
- `Update recommendation`
- `Submit board suggestion`

That matches the moderation reality and reduces the expectation of immediate publication.

## Product Requirements Interpreted

Based on your request, the desired behavior is:

1. Any page that hosts a whiteboard should be able to use the same save flow.
2. Public visitors should be able to submit recommendations to GitHub.
3. If the same person submits again for the same board, the app should update their existing recommendation instead of creating a new unrelated submission.
4. Local exports should use a timestamped filename:
   `board_name_YYYY-MM-DD_HH-mm-ss.canvas`
5. File history should be versioned sensibly without filling the repo with unrelated duplicate snapshots.

## Recommended Architecture

### Frontend

Keep the site static.

The frontend should:

- render the board UI
- load the canonical board file
- let the user edit locally first
- initiate GitHub auth
- submit or update a recommendation PR

### Tiny Auth Exchange Backend

Use a minimal serverless endpoint only for secure OAuth code exchange.

Good options:

- Cloudflare Worker
- Vercel Function
- Netlify Function

This endpoint must hold:

- `GITHUB_CLIENT_ID`
- `GITHUB_CLIENT_SECRET`

The frontend must never contain the client secret.

### GitHub REST API

For recommendation submissions, use:

- fork creation
- branch creation
- file read/write
- pull request create/list

### Recommended v1 submission model

For each board and GitHub user, keep one stable recommendation branch in the user's fork.

Example branch name:

- `recommend-braindump-evrenucar`
- `recommend-open-quests-janedoe`

If that branch already has an open PR to upstream, reuse it.

That gives you:

- additive updates instead of independent submissions
- clean PR history
- easy moderation
- natural version control through git commits on the same branch

## Reusable Multi-Board Design

The current implementation is tied directly to `braindump.html`.

Refactor the whiteboard so any page can host it through board metadata.

### Introduce a board config model

Each board should have:

- `boardSlug`
- `boardTitle`
- `sourcePath`
- `storageKey`
- `pagePath`
- `allowRecommendations`

Example:

```json
{
  "boardSlug": "braindump",
  "boardTitle": "Braindump",
  "sourcePath": "content/boards/braindump/current.canvas",
  "storageKey": "board:braindump",
  "pagePath": "braindump.html",
  "allowRecommendations": true
}
```

### Render boards by configuration instead of hardcoding one page

Instead of relying on `id="braindump-viewport"` and one hardcoded source file, move toward a generic initializer:

- HTML container carries board metadata via `data-*`
- JS bootstraps the board from those attributes
- save/export/import logic reads the current board config

That makes it possible to embed the same board system in multiple generated pages.

## File Strategy

### Canonical in-repo board file

Use one stable canonical file per board, for example:

- `content/boards/braindump/current.canvas`
- `content/boards/quest-map/current.canvas`

Why stable instead of timestamped in the repo:

- the live board needs one predictable fetch target
- PR diffs stay readable
- repeated recommendations can update one file
- git already provides version history

### Exported downloads

Downloads should be timestamped for user convenience:

- `braindump_2026-04-19_14-37-05.canvas`

Rules:

- sanitize board title into a safe filename
- use local browser time or UTC consistently
- keep `.canvas` extension

### Optional later snapshot archive

If you want archival snapshots later, add an optional export path like:

- `content/boards/braindump/history/2026-04-19_14-37-05.canvas`

But do not use snapshot-per-save as the main recommendation flow. That would create the "new independent one every time" behavior you explicitly do not want.

## Recommendation Submission Flow

### 1. User edits locally first

The board should continue to work without auth.

Local state stays useful for:

- drafts
- anonymous experimentation
- offline edits
- recovery if the GitHub flow fails

### 2. User clicks `Send recommendation`

The app checks for GitHub auth.

If not authenticated:

- redirect to GitHub OAuth

If authenticated:

- continue with submission

### 3. OAuth callback

The callback page should:

- validate `state`
- exchange code for token through the serverless function
- preferably use PKCE
- keep the token in memory or `sessionStorage`, not `localStorage`

### 4. Identify the user

Use GitHub API to fetch:

- login
- user id

This becomes part of the stable branch naming and PR lookup.

### 5. Ensure fork exists

If the user does not already have a fork of the upstream repo:

- create it

Fork creation is asynchronous, so the flow should poll until the fork is ready.

### 6. Ensure stable recommendation branch exists

Branch naming rule:

- one branch per `(boardSlug, githubLogin)`

Example:

- `recommend-braindump-janedoe`

If it does not exist:

- create it from upstream `main`

If it already exists:

- reuse it

### 7. Commit updated board file to the branch

Write the board file to the stable canonical path for that board.

Example:

- `content/boards/braindump/current.canvas`

Commit message example:

- `Recommend update to Braindump`
- `Update recommendation for Braindump`

### 8. Reuse or create the PR

Check if there is already an open PR from:

- `userfork:recommend-braindump-janedoe`

to:

- `upstream:main`

If yes:

- do not create a second PR
- return that PR URL
- tell the user their recommendation was updated

If no:

- create a new draft PR

Suggested PR title:

- `Recommend update: Braindump`

Suggested PR body should include:

- board title
- submitter login
- source page URL
- timestamp
- optional user note

Draft PR is the right default because approval will not be instant.

## Conflict Handling

Do not attempt multi-user semantic merging of `.canvas` content in v1.

That is too risky for a first implementation.

Instead:

- store the upstream base commit SHA when the board is loaded
- on submit, compare current upstream SHA with that base SHA

If unchanged:

- proceed normally

If changed:

- show a warning that the live board changed since the user loaded it
- let them choose:
  - reload latest board
  - continue submitting anyway

For v1, continuing anyway is acceptable because the recommendation lives on the user's branch and review happens in the PR.

## Recommended Implementation Phases

### Phase 0: Board infrastructure cleanup

- move canonical file to board-based path
- add board metadata
- make export filename timestamped
- keep local save/import/export working

Deliverable:

- one reusable board can still fully work offline/local-first

### Phase 1: Single-board recommendation MVP

- GitHub OAuth app
- tiny auth exchange function
- callback page
- fork-based PR flow for `braindump`
- one stable recommendation branch per user
- one stable PR per user per board

Deliverable:

- Braindump can submit/update a recommendation PR

### Phase 2: Generalize to multiple whiteboards

- convert page-specific bootstrap into reusable initializer
- board config registry
- reusable toolbar/actions

Deliverable:

- any generated page can host a board and submit recommendations

### Phase 3: Moderation and conflict polish

- base SHA conflict warning
- optional draft note from submitter
- clearer submission state in UI
- better PR body metadata

### Phase 4: Optional server-side hardening

If frontend token handling feels too exposed, move GitHub write operations into the serverless function so the browser never directly calls GitHub with the access token.

That increases backend surface area, so it should be a later phase unless security constraints demand it immediately.

## Recommendation On OAuth App vs GitHub App

### Recommended for v1

Use a GitHub OAuth App plus a tiny code exchange backend.

Reason:

- simpler to ship
- fits the current static site constraint
- aligns with "use their own GitHub account"

### Better long-term alternative

Consider a GitHub App later if you want:

- finer-grained permissions
- shorter-lived tokens
- stronger production controls

But it is more setup than the repo currently needs.

## Final Recommendation

Implement the recommendation flow as:

- local-first board editing
- GitHub login only when submitting
- fork-based PRs for public users
- one stable branch and one stable open PR per user per board
- one stable canonical board file path per board in the repo
- timestamped filenames only for user downloads

That satisfies all of your requirements without fighting GitHub's permission model, and it fits this repo's current static architecture with the smallest realistic expansion.
