# Security Audit 20260429 041336

Date: 2026-04-29
Repository: `proto_website`
Scope: static site, Cosmoboard/Braindump browser runtime, local preview/dev servers, build/sync scripts, GitHub Actions, dependencies, and existing security/access planning.

## Executive Summary

This repository is suitable for a personal local-first prototype, but it is not yet safe for untrusted board files, sensitive private content, or a LAN-exposed save server. The main security risk is the combination of:

- local preview/dev servers that expose unauthenticated write APIs,
- rendering paths that trust canvas, markdown, link, SVG, and JSON content,
- no CSP or equivalent browser containment,
- cleartext localStorage persistence for board data.

No hard-coded production secrets were found, and `npm audit` reports zero known vulnerabilities for the installed dependency tree. The highest-priority work is to lock down the local write server and eliminate stored-XSS paths in `JavaScript/braindump.js`.

## Threat Model Used

This audit considered these realistic attackers:

- a malicious website visited while the preview server is running locally,
- another device on the same LAN when the preview server is bound to all interfaces,
- a malicious or compromised `.canvas`, markdown, bundle, or imported JSON file,
- compromised public Notion/content data flowing into the generated static site,
- an untrusted PR author or branch with workflow execution access.

Out of scope for this pass:

- malware or a malicious browser extension on the user's machine,
- state-level adversaries,
- full cryptographic design review of future encrypted sharing phases.

### S-01 - Unauthenticated LAN-Reachable Write APIs

`scripts/preview-server.mjs` exposes write endpoints for boards, markdown, and assets, then listens on `0.0.0.0`:

- `scripts/preview-server.mjs:505-516`
- `scripts/preview-server.mjs:568-573`

`scripts/dev-server.mjs` exposes a save endpoint too and calls `server.listen(PORT)` without a loopback-only hostname:

- `scripts/dev-server.mjs:26-38`
- `scripts/dev-server.mjs:70-72`

Impact:

- Any device that can reach the preview port can overwrite board files, markdown files, and uploaded assets.
- A malicious website can still trigger "simple" cross-origin POSTs to localhost/LAN services even without reading the response. CORS does not protect state-changing endpoints by itself.
- Existing endpoint tests confirm the write path works without authentication: `tests/preview/preview-save-endpoint.test.mjs` passed.

Recommended fix:

- Bind to `127.0.0.1` by default.
- Require an explicit `HOST=0.0.0.0` or `--lan` flag for LAN access, with a warning.
- Generate a random per-server token on startup and require it for all mutating requests.
- Reject unsafe `Origin` and `Host` headers. Allow only `http://127.0.0.1:<port>`, `http://localhost:<port>`, and any explicitly enabled LAN origin.
- Enforce `Content-Type` for JSON endpoints and reject form/text submissions unless explicitly supported.

### S-02 - Broad Sidecar Writes From `/api/save-board`

`handleSaveBoard` writes the main canvas, then iterates over markdown nodes and writes each `_rawMarkdown` to `node.file`:

- `scripts/preview-server.mjs:138-162`

For absolute-like paths, `node.file` is resolved under `rootDir`:

- `scripts/preview-server.mjs:149-153`

The containment check only verifies `sidecarPath.startsWith(rootDir)`:

- `scripts/preview-server.mjs:155`

Impact:

- A malicious canvas payload can write any `.md` file under the repo root, not only the board's markdown folder.
- The string-prefix containment pattern is fragile. A sibling path whose name starts with the repo path prefix can pass on some path shapes. Use `path.relative` based containment instead.
- Because S-01 leaves the endpoint unauthenticated, this becomes a remote integrity issue whenever the preview server is reachable.

Recommended fix:

- Do not trust `node.file` as an arbitrary write target.
- Resolve all markdown sidecars through the same board-local helper used by `/api/save-markdown`.
- Restrict sidecars to `content/boards/<slug>/markdown/*.md` or another explicit board-owned directory.
- Replace every `startsWith(rootDir)` path check with a helper:

```js
function isInside(base, target) {
  const relative = path.relative(base, target);
  return relative === "" || (!relative.startsWith("..") && !path.isAbsolute(relative));
}
```

### S-03 - Stored XSS In Link Preview Nodes

`renderLinkNode` builds preview-mode HTML by interpolating node-controlled values directly:

- `JavaScript/braindump.js:3645-3655`

Unsafe fields include:

- `nodeObj.image` in an `<img src>` attribute,
- `nodeObj.title` inside `<h3>`,
- `nodeObj.description` inside `<p>`,
- `nodeObj.url` in an `<a href>` attribute and text body.

Impact:

- A malicious imported canvas or bundle can execute script when the board renders.
- A compromised link-preview response can poison persisted node fields.
- This risk is amplified by cleartext localStorage persistence and unauthenticated local save APIs.

Recommended fix:

- Build link preview DOM with `document.createElement`, `textContent`, and `setAttribute`, not template `innerHTML`.
- Add a central `safeUrl()` helper that only permits `http:`, `https:`, and explicitly approved local relative URLs.
- Reject `javascript:`, `data:text/html`, `file:`, and malformed URLs for clickable links and iframes.
- Treat third-party preview metadata as untrusted input.

### S-04 - Unsafe Markdown Links And Inline SVG Draw Nodes

Markdown rendering escapes `<`, `>`, and `&`, then injects link URLs into `href` attributes without escaping quotes or validating protocols:

- `JavaScript/braindump.js:3872-3883`
- `JavaScript/braindump.js:3939-3951`
- `JavaScript/braindump.js:4093-4098`
- `JavaScript/braindump.js:4102-4107`

Drawing nodes are stored as SVG strings in text nodes and inserted directly:

- `JavaScript/braindump.js:6877-6880`

Impact:

- Markdown like `[x](javascript:alert(1))` creates a clickable script URL.
- Markdown URLs containing quotes can break out of `href`.
- Imported text nodes containing `<svg ...>` can inject active SVG content or event handlers.

Recommended fix:

- Use a markdown parser/sanitizer pair, or keep the custom renderer but generate DOM nodes rather than HTML strings.
- Escape attribute values, not only text nodes.
- Add protocol allowlisting for all generated links.
- Store drawings as structured geometry, or sanitize SVG through a strict SVG allowlist before rendering.

### S-05 - Missing CSP And Security Headers

The preview server sends only content type/length for static files:

- `scripts/preview-server.mjs:37-43`
- `scripts/preview-server.mjs:557-560`

The generated site includes inline scripts and JSON-LD script tags:

- `scripts/build-site.mjs:440-506`

Impact:

- Any XSS sink has full script execution capability.
- SVG, iframe, markdown, and link preview surfaces have no browser-level containment.
- There is no `X-Content-Type-Options: nosniff`, `Referrer-Policy`, `Permissions-Policy`, or frame policy on the local preview server.

Recommended fix:

- Start with `Content-Security-Policy-Report-Only` in preview builds.
- Move inline navigation state code into `JavaScript/site.js`, or hash it for CSP.
- Target a first CSP shape similar to:

```txt
default-src 'self';
script-src 'self';
style-src 'self' 'unsafe-inline';
img-src 'self' data: blob: https:;
font-src 'self';
connect-src 'self' https://www.youtube.com https://api.microlink.io;
frame-src 'self' https://www.youtube-nocookie.com;
object-src 'none';
base-uri 'self';
frame-ancestors 'self';
```

Then tighten `connect-src`, `img-src`, and `frame-src` as link-preview and embed policies mature.

### S-06 - Cleartext Board State In localStorage

Board state is persisted directly to localStorage:

- `JavaScript/braindump.js:1312-1318`
- `JavaScript/braindump.js:1322-1325`

The security planning note already says the product direction is encryption-first and public-host-safe:

- `.agents/brainstorming_planning/security_and_access.md`

Impact:

- Any XSS vulnerability gets the full board state.
- Browser profiles, backups, shared computers, or extensions can read the data.
- This does not match the long-term "safe by default" product goal.

Recommended fix:

- Do not market current boards as private or encrypted.
- For the next security phase, store sensitive state in encrypted IndexedDB using a user-held key.
- Keep a clear "prototype local cleartext" label until encryption-at-rest exists.
- Prefer the planned Phase B path: passphrase plus recovery file, then age-style envelopes.

### S-07 - Automatic Link Preview Privacy Leak

When a link node renders, the client fetches third-party preview metadata:

- YouTube oEmbed: `JavaScript/braindump.js:935-945`
- Microlink: `JavaScript/braindump.js:953-962`

Impact:

- Pasted private URLs, local URLs, draft URLs, or internal references can be sent to third-party services.
- The fetch happens as a rendering behavior, not an explicit user-approved action.

Recommended fix:

- Disable third-party preview fetching by default for private/local boards.
- Add a one-time consent setting: "Fetch public link previews through external services".
- Never send `localhost`, `127.0.0.0/8`, private LAN ranges, `file:`, `blob:`, `data:`, or non-HTTP(S) URLs to preview services.
- Cache previews locally and show the source service in the UI.

### S-08 - Iframe Sandbox Is Too Permissive And YouTube Privacy Regression Exists

Live link embeds use:

- `sandbox="allow-scripts allow-same-origin allow-popups allow-forms allow-presentation"` at `JavaScript/braindump.js:3598`

App iframes use:

- `sandbox="allow-scripts allow-same-origin allow-popups allow-forms"` at `JavaScript/braindump.js:3723`

The current `getYouTubeEmbedUrl` returns `https://www.youtube.com/embed/...`:

- `JavaScript/braindump.js:884-888`

The existing regression test expects `youtube-nocookie.com`, and it currently fails:

- `tests/features/youtube-live-embed.test.mjs`

Impact:

- `allow-scripts` plus `allow-same-origin` removes much of the value of sandboxing for same-origin content.
- Arbitrary live embeds get broad powers.
- YouTube embeds use the tracking-heavy domain despite a test expecting the privacy-preserving domain.

Recommended fix:

- Use `youtube-nocookie.com` for YouTube embeds.
- Add `referrerpolicy="strict-origin-when-cross-origin"` to runtime-generated iframes.
- For arbitrary external embeds, remove `allow-same-origin` unless the source is explicitly trusted.
- Separate policies for trusted same-origin apps, trusted video providers, PDFs, and arbitrary web pages.

### S-09 - Weak Size, Timeout, And Rate Controls

`handleSaveBoard` and `handleSaveMarkdown` accumulate request bodies into strings without a size cap:

- `scripts/preview-server.mjs:73-82`
- `scripts/preview-server.mjs:227-235`

`handleSaveAsset` has a cap, but it is high for an unauthenticated LAN service:

- `scripts/preview-server.mjs:304-305`

Server-side metadata fetch has a response-data cap but no explicit timeout and recursively follows redirects:

- `scripts/preview-server.mjs:422-450`

Notion asset and preview-image fetches also lack explicit size and timeout controls:

- `scripts/sync-notion.mjs:682-698`
- `scripts/sync-notion.mjs:719-725`

Impact:

- A local or LAN attacker can consume memory, disk, or network resources.
- CI/site sync can be slowed or filled by unexpectedly large remote assets.
- Redirect chains and slow responses can hang local preview operations.

Recommended fix:

- Add request-size caps to JSON endpoints.
- Lower unauthenticated asset caps, or require an auth token before accepting large assets.
- Add `AbortController` timeouts to all external fetches.
- Limit redirect depth.
- Validate content type and content length before downloading remote assets.

### S-10 - GitHub Actions Uses Broad Write Permission

The workflow grants `contents: write` at workflow scope:

- `.github/workflows/build-site.yml:16-17`

It runs on `pull_request`, `push`, `workflow_dispatch`, and hourly `schedule`:

- `.github/workflows/build-site.yml:3-9`

The commit step is guarded to avoid PR writes:

- `.github/workflows/build-site.yml:37-48`

Impact:

- The current guard is helpful, but the token scope is broader than the build/test steps need.
- Same-repository PRs or future workflow edits have more permission than necessary.

Recommended fix:

- Set workflow-level permissions to `contents: read`.
- Split the "commit generated content" operation into a separate job or step with `contents: write` only on `main` push/schedule.
- Consider pinning third-party actions by commit SHA for stronger supply-chain control.

### S-11 - SVG Uploads Are Allowed

The asset upload denylist blocks executable extensions but allows `.svg`:

- `scripts/preview-server.mjs:276-289`
- `scripts/preview-server.mjs:314-320`

The static server serves SVG as `image/svg+xml`:

- `scripts/preview-server.mjs:14-27`

Impact:

- SVG is active XML and can carry script/event payloads in some render contexts.
- Current image rendering uses `<img>`, which is safer, but direct opening, iframe usage, or future render paths can execute more of the SVG surface.

Recommended fix:

- Either block SVG uploads for now or sanitize SVG with a strict allowlist.
- Add `X-Content-Type-Options: nosniff`.
- Do not render uploaded SVG in iframes or as inline SVG unless sanitized.

### S-12 - Localhost/Blob URLs Are Persisted In Board Content

Committed board content includes local-only URLs:

- `content/boards/braindump/current.canvas` contains `blob:http://127.0.0.1:4173/...`.
- `content/boards/braindump/current.canvas` and markdown sidecars contain `http://localhost:4173/cosmoboard.html`.
- `content/boards/cosmoboard/current.canvas` contains `http://127.0.0.1:4173/content/boards/cosmoboard/participant-information.pdf`.

Impact:

- Public readers may attempt to load resources from their own localhost.
- Portability suffers because state depends on a previous local server session.
- This can become a browser-side localhost request trigger if content is embedded live.

Recommended fix:

- Normalize saved board URLs to repo-relative paths where possible.
- Reject or warn before committing `blob:`, `file:`, `localhost`, and `127.0.0.1` references.
- Add a content hygiene test for board files.

## Recommended Remediation Order

1. Lock down local servers.
   - Bind to loopback by default.
   - Require a per-session token for all POST endpoints.
   - Add origin/host checks.
   - Add body-size limits.

2. Fix repo write boundaries.
   - Replace `startsWith(rootDir)` checks.
   - Restrict markdown sidecars to board-owned markdown directories.
   - Fix `/api/save-markdown` default path to match the `markdown/` subfolder test expectation, or update the test if the board-root path is intentional.

3. Remove stored-XSS surfaces.
   - Rewrite link preview rendering without `innerHTML`.
   - Sanitize markdown links and protocols.
   - Stop rendering raw SVG strings from imported board data.

4. Add browser defense in depth.
   - Add CSP in report-only mode.
   - Add `nosniff`, `Referrer-Policy`, and `Permissions-Policy`.
   - Move or hash inline scripts.

5. Tighten iframe and preview privacy.
   - Restore `youtube-nocookie.com`.
   - Add iframe `referrerpolicy`.
   - Remove `allow-same-origin` for arbitrary embeds.
   - Disable third-party preview fetches by default for private/local boards.

6. Align with encryption-first planning.
   - Keep current cleartext mode explicitly labeled as prototype/local-only.
   - Implement encrypted local persistence before storing sensitive content.
   - Use the planned passphrase/recovery-file phase before public encrypted sharing.

7. Harden CI.
   - Reduce default permissions to `contents: read`.
   - Grant write only in the generated-content commit path.
   - Add a security hygiene check for local URLs and unsafe protocols in board files.

## Suggested Regression Tests

- Cross-origin POST without token to `/api/save-board` returns 403.
- POST with bad `Origin` returns 403.
- `/api/save-board` markdown sidecar cannot write outside `content/boards/<slug>/markdown/`.
- `node.file` with `../`, absolute-like paths, and sibling-prefix paths is rejected.
- Link node title/description/url render as text and safe attributes.
- Markdown `javascript:` links are inert or dropped.
- Markdown URLs containing quotes cannot create extra attributes.
- Imported SVG text nodes are sanitized or rejected.
- `getYouTubeEmbedUrl()` always returns `youtube-nocookie.com`.
- Board content tests reject committed `blob:`, `file:`, `localhost`, and `127.0.0.1` references unless explicitly allowlisted.
